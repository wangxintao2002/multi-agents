# 附件 SOW

# 大规模多智能体协同调度和高效通信关键技术研究（保护用户数据隐私）任务书

## 1. 研究背景

随着 Claude Code、OpenClaw、Hermes Agent 等智能体应用快速兴起，Agent 正在从单轮问答工具演进为能够调用工具、读写文件、操作系统应用并协同完成长程任务的执行主体。未来复杂任务将从单个助手扩展为可编排的智能体集群：广域信息获取与方案探索需要并行覆盖不同资料、工具与假设空间；长流程生产任务需要把规划、执行、验证、审查、监控、合规等职责长期分离，形成多阶段流水线；评测、数字运营与大规模仿真类任务需要同时维护大量长期运行、各自持有独立状态与工作区的 Agent。近期 AgentScope 等大规模多智能体仿真工作采用分布式机制支撑大量 Agent 并行交互，Anthropic 生产级多 Agent 研究系统通过主控 Agent 并行派生多个子 Agent 提升开放式任务覆盖度，A2A/MCP 等协议也显示智能体正从封闭应用走向跨系统协作。由此，Agent 成为带有身份、权限、工具、预算、上下文与副作用边界的运行时实体；系统需要像操作系统调度进程、云平台调度容器一样，对协作中的 Agent 进行调度与协同。

当系统从单个助手扩展为多 Agent 集群时，真正决定运行时价值的不是 Agent 数量，而是 Agent 之间是否耦合。当 Agent 之间同时出现两类耦合，运行时治理成为刚需：其一是争抢有限的共享基础设施，其二是 Agent 间任务依赖（规划→执行→验证的结果传递与等待）。在工具型、执行型 Agent 场景中，运行时真正拥有并能够调度的有限资源不仅包括 LLM token 或 API 预算，还包括本地或远端执行沙箱（Docker、microVM、代码解释器）、浏览器自动化实例、搜索/数据库/内部工具 API 配额、本机 CPU/内存/磁盘 IO、每个 Agent 的工作区与文件系统状态、上下文窗口以及多任务并发度。这些资源并非由模型服务商统一接管，而是由客户端运行时、企业私有部署或混合云平台实际管理，是大规模 Agent 系统吞吐与稳定性的关键瓶颈。

现有多智能体框架多把 Agent 逻辑、工作区状态与活沙箱执行资源绑定在一起：一个 Agent 从任务开始到结束长期占用一个沙箱，即使它正在等待模型返回、等待 CI、等待其他 Agent 产出或等待人工确认，也不会释放沙箱 slot、内存、文件句柄和工作区资源。在并发任务规模上升后，这种 task-lease 式绑定容易造成沙箱队列拥塞、容器冷启动放大、工具配额争抢、闲置资源被长期占用以及任务流尾延迟上升。另一方面，若没有运行时维护的状态边界与资源租约机制，简单复用活沙箱又会引入依赖污染、文件污染和安全隔离问题。因此，大规模工具型 Agent 运行时需要把 Agent 逻辑状态、工作区状态和活执行资源解耦：Agent 可以在等待外部事件时挂起，工作区状态可以通过快照、挂载或受控持久化保留，活沙箱只在执行 shell、测试、构建、浏览器操作等步骤时被租用。

通信侧也存在类似的运行时治理问题。多 Agent 通信若采用 all-to-all 自由文本广播、共享 scratchpad 或中心 manager 转发，在 Agent 数量、任务轮次与工具输出增加后会造成消息扇出、上下文膨胀、信息污染与中心节点瓶颈。上下文窗口本身也是受限资源，消息投递策略实质上是上下文预算的分配器。运行时需要同时管理计算资源、工具资源、工作区状态与上下文资源，才能在大规模协作中保持吞吐、隔离性和可观测性。

AgentOS 的目标，是把协作中的 Agent 纳入运行时约束下的可调度计算资源体系：Agent 是可调度实体，消息是可路由事件，沙箱、工具、工作区与上下文窗口是受控资源。运行时需要在统一控制面内提供图式协作调度、多资源租约、状态挂起与恢复、沿边数据流与结构性上下文隔离等基础能力。

## 2. 研究内容

本项目围绕耦合型多智能体协作中的执行控制问题，研究一组面向 Agent 的运行时内核机制，把调度与通信从提示词约定下沉为统一运行时提供的系统级服务。整体采用“应用层编译 + 运行时执行”的分层架构：用户用自然语言描述任务，应用层的主 Agent 把任务编译成一张静态或半静态工作流图（workflow），提交给 AgentOS 运行时；运行时把图节点物化为 Agent，用调度、资源租约、状态管理与通信支撑这些 Agent 运行。研究内容主要包括调度机制与通信机制，并明确相关技术边界与创新定位。

### 2.1 调度机制：统一事件模型与多资源状态感知调度

与传统一次性模型请求不同，Agent 的执行过程由多轮推理、工具调用、消息交互、资源等待与阶段切换组成，且大量时间消耗在等待外部慢操作（等模型生成、等测试跑完、等人审批）。若缺少运行时调度机制，只能依赖周期性唤起或固定 workflow 维持等待逻辑，造成 token 浪费、状态漂移与异常任务堆积。

项目拟以统一事件模型作为运行时的核心，所有会发生的事——模型调用完成、工具跑完、外部 CI 回调、定时器到期、资源可用、消息到达、超时、取消、“人点了批准”——全部规范化成同一类 Runtime Event，进入同一个控制队列。在此之上建立三层调度结构：Workflow 决定一组最多几个 Agent 同时运行、组级预算多少；Agent 是被调度的主体，在就绪队列中排队、有优先级、能挂起与唤醒；Step（一次节点执行）是最小派发单元，调度器每次只派给 Worker 一个 step，原子不可分。step 内部不被抢占（仅软超时兜底），模型与工具调用作为指令从 step 中“返回出来”、交给循环外的网关异步执行。

在统一事件模型之上，本项目进一步研究多资源状态感知调度机制。运行时把 Agent 逻辑状态、工作区状态和活执行资源拆分管理：Agent 逻辑状态包括当前任务阶段、消息、未完成 step、预算与等待条件；工作区状态包括代码仓库、依赖安装结果、生成文件、测试产物与快照引用；活执行资源包括沙箱 slot、浏览器实例、工具 API 并发额度、CPU/内存/磁盘 IO 等。调度器通过资源租约机制控制 step 是否能够进入执行阶段：当 step 需要 shell、测试、构建、浏览器或外部工具时，必须先获得相应资源租约；当 step 转入 LLM 等待、外部事件等待或人工审批等待时，运行时可挂起 Agent 并释放活执行资源，等待资源可用事件或外部事件再次唤醒。

调度机制包括以下关键能力：一是资源感知准入控制，根据沙箱池容量、工具配额、组级预算和任务优先级决定 workflow 是否进入运行；二是 task-lease 与 exec-lease 两种运行模式对比，task-lease 作为朴素基线让 Agent 在整个任务期间独占沙箱，exec-lease 作为优化模式只在实际执行命令时租用活沙箱；三是沙箱池化与亲和性复用，对相同仓库、相同依赖或同一 workflow 的后续 step 优先复用热环境或快照，降低冷启动与依赖安装成本；四是挂起/唤醒与状态恢复，在 Agent 等待慢事件时释放活资源，通过工作区快照、挂载目录或可恢复状态继续执行；五是资源可观测性，记录沙箱等待时间、沙箱利用率、闲置占用时间、工具配额等待、flow time 与失败原因，为调度策略优化提供依据。

### 2.2 通信机制：沿边数据流与结构性上下文隔离

通信机制面向多 Agent 协作中的消息扇出、上下文膨胀与权限隔离问题。现有多智能体系统常采用 all-to-all 广播、共享 scratchpad 或中心 manager 转发，在 Agent 数量与任务轮次上升后会导致大量无关消息进入每个 Agent 上下文、中心节点承担过重的汇总转发压力、消息来源与处理状态难以追踪。

本项目拟建立运行时管理的 Agent 通信与上下文命名空间，由运行时维护 Agent 身份、通信通道、主题订阅、收件箱、消息记录、投递状态和资料引用。借鉴受控共享缓冲区和事件总线的思想，通信层将消息、状态摘要和资料引用作为运行时对象管理，避免把公共对话历史直接暴露给所有 Agent。通信通道覆盖点对点、主题、群组和运行时控制等协作关系；Agent 发送消息时提交结构化元数据、短摘要和资料引用。运行时根据接收者、主题、角色、权限、任务阶段和过滤条件选择性投递，避免无关消息进入不相关 Agent 的上下文。

通信机制同时服务于资源调度。上下文窗口被视为一种受限资源，运行时根据任务阶段、Agent 角色和依赖边选择性注入消息、摘要与引用，避免把无关工具输出、其他 Agent 私有状态或过期中间结果放入下游上下文。对大文件、测试日志、网页内容和代码片段，运行时优先传递可追踪引用和结构化摘要，仅在接收 Agent 真正需要时按需展开。由此，通信层不仅降低消息扇出和上下文污染，也为调度层提供明确的依赖边、等待条件和唤醒事件。

### 2.3 相关技术边界与创新定位

本项目借鉴 durable execution、persistent sandbox 与 agent sandbox checkpoint/restore 等方向的已有基础。Temporal、Dapr、Restate、LangGraph 等系统证明了工作流或 Agent 逻辑状态可以通过事件日志、checkpoint 或 durable step 从具体 worker 进程中解耦；E2B、Modal、Daytona、OpenKruise Agents 等平台证明了沙箱可以通过 pause/resume、snapshot、fork、filesystem snapshot 等方式管理长期会话状态；Crab 等 Agent sandbox checkpoint/restore 研究进一步指出，仅保存聊天历史无法覆盖文件系统、进程和运行时副作用，OS 可见状态需要进入恢复模型。

本项目的创新点不在于单独发明 checkpoint、沙箱快照或工作流恢复，而在于面向大规模多 Agent workload，把 Agent 逻辑状态、工作区状态、活沙箱执行资源、工具配额和上下文预算统一纳入运行时调度模型，在沙箱与工具资源受限的真实执行场景下提升任务吞吐、资源利用率与上下文隔离性。

### 3.2 验收指标

#### 3.2.1 软件验收条件

**一套面向大规模多智能体的运行时**

- 交付件形态：项目原型及源码
- 运行环境：基于业界主流服务器和 Linux 操作系统
- 功能描述：支持拆解用户任务并组织成多个 Agent 的协作流程由 Agent 运行时执行，Agent 运行时负责调度各 Agent 的调度顺序、协调它们的共享资源并在它们之间传递数据与结果。
- 验收标准：
  - 阶段 1.1：Agent 运行时可启动，能把多个 Agent 作为统一管理的对象来驱动运行
  - 阶段 1.2：支持多个 Agent 按任务依赖关系协作，上游完成后结果传给下游并触发调度运行
  - 阶段 2.1：支持沙箱、工具 API、浏览器实例、工作区状态和上下文窗口等资源的统一建模与租约管理
  - 阶段 2.2：支持 Agent 挂起与唤醒；Agent 等待模型、外部事件或人工审批时，可释放活沙箱执行资源，并在条件满足后恢复执行
  - 阶段 2.3：支持 Agent 间通信以及上下文隔离，避免无关信息互相干扰
  - 阶段 2.4：支持多个任务流程同时运行，并由 Agent 运行时在多个并行 Agent 之间协调调度、通信和共享资源
  - 性能指标：在沙箱或工具资源受限的并发工具型 Agent workload 下，相比 naive task-lease FCFS 系统，任务吞吐指标提升 10%；记录并分析 sandbox slot wait time、sandbox utilization、idle-held sandbox time、flow time p50/p95、工具配额等待时间和任务完成率
  - 通信指标：相比 all-to-all 广播、共享 scratchpad 或中心 manager 转发等 naive 通信方式，无关上下文注入率降低 10%
