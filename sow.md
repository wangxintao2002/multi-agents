# 大规模多智能体协同调度和高效通信关键技术研究（保护用户数据隐私）任务书

## 1. 研究背景

随着 Claude Code、OpenClaw、Hermes Agent 等智能体应用快速兴起，Agent 正在从单轮问答工具演进为能够调用工具、读写文件、操作系统应用并协同完成长程任务的执行主体。未来复杂任务将从单个助手扩展为可编排的智能体集群：广域信息获取与方案探索需要并行覆盖不同资料、工具与假设空间；长流程生产任务需要把规划、执行、验证、审查、监控、合规等职责长期分离，形成多阶段流水线；评测、数字运营与大规模仿真类任务需要同时维护大量长期运行、各自持有独立状态与工作区的 Agent。近期 AgentScope 等大规模多智能体仿真工作采用分布式机制支撑大量 Agent 并行交互，Anthropic 生产级多 Agent 研究系统通过主控 Agent 并行派生多个子 Agent 提升开放式任务覆盖度，A2A/MCP 等协议也显示智能体正从封闭应用走向跨系统协作。由此，Agent 成为带有身份、权限、工具、预算、上下文与副作用边界的运行时实体；系统需要像操作系统调度进程、云平台调度容器一样，对协作中的 Agent 进行调度与协同。

当系统从单个助手扩展为多 Agent 集群时，真正决定运行时价值的不是 Agent 数量，而是 Agent 之间是否耦合。当 Agent 之间同时出现两类耦合，运行时治理成为刚需：争抢有限的共享基础设施）、Agent 间任务依赖（规划→执行→验证的结果传递与等待）。在现有多智能体运行方式下，尤其当多个耦合 Agent 并发协作时，本项目拟解决两个系统级问题。其一，长程 Agent 缺少操作系统级调度原语：Agent 在等待邮件、文件变化、工具返回、CI 结果、其他 Agent 阶段性产出或人工确认时，只能依赖周期性唤起模型检查条件、workflow 硬编码或上下文记忆维持等待逻辑，导致无效推理、token 浪费与状态漂移。其二，多 Agent 通信若采用 all-to-all 自由文本广播、共享 scratchpad 或中心 manager 转发，在 Agent 数量、任务轮次与工具输出增加后会造成消息扇出、上下文膨胀、信息污染与中心节点瓶颈。

AgentOS 的目标，是把协作中的 Agent 纳入运行时约束下的可调度计算资源体系：Agent 是可调度实体，消息是可路由事件，工具与基础设施是受控资源。运行时需要在统一控制面内提供图式协作调度、沿边数据流与结构性上下文隔离等基础能力。

## 2. 研究内容

本项目围绕耦合型多智能体协作中的执行控制问题，研究一组面向 Agent 的运行时内核机制，把调度与通信从提示词约定下沉为统一运行时提供的系统级服务。整体采用“应用层编译 + 运行时执行”的分层架构：用户用自然语言描述任务，应用层的主 Agent 把任务编译成一张静态工作流图（workflow），提交给 AgentOS 运行时；运行时把图节点物化为 Agent，用调度与通信支撑这些 Agent 运行。研究内容包括两部分：调度机制、通信机制。

### 5.1 调度机制：统一事件模型

与传统一次性模型请求不同，Agent 的执行过程由多轮推理、工具调用、消息交互、资源等待与阶段切换组成，且大量时间消耗在等待外部慢操作（等模型生成、等测试跑完、等人审批）。若缺少运行时调度机制，只能依赖周期性唤起或固定 workflow 维持等待逻辑，造成 token 浪费、状态漂移与异常任务堆积。

项目拟以统一事件模型作为运行时的核心，所有会发生的事——模型调用完成、工具跑完、外部 CI 回调、定时器到期、资源可用、消息到达、超时、取消、“人点了批准”——全部规范化成同一类 Runtime Event，进入同一个控制队列。在此之上建立三层调度结构：Workflow 决定一组最多几个 Agent 同时运行、组级预算多少；Agent 是被调度的主体，在就绪队列中排队、有优先级、能挂起与唤醒；Step（一次节点执行）是最小派发单元，调度器每次只派给 Worker 一个 step，原子不可分。step 内部不被抢占（仅软超时兜底），模型与工具调用作为指令从 step 中“返回出来”、交给循环外的网关异步执行。

### 5.2 通信机制：沿边数据流与结构性上下文隔离

通信机制面向多 Agent 协作中的消息扇出、上下文膨胀与权限隔离问题。现有多智能体系统常采用 all-to-all 广播、共享 scratchpad 或中心 manager 转发，在 Agent 数量与任务轮次上升后会导致大量无关消息进入每个 Agent 上下文、中心节点承担过重的汇总转发压力、消息来源与处理状态难以追踪。

本项目拟建立运行时管理的 Agent 通信与上下文命名空间，由运行时维护 Agent 身份、通信通道、主题订阅、收件箱、消息记录、投递状态和资料引用。借鉴受控共享缓冲区和事件总线的思想，通信层将消息、状态摘要和资料引用作为运行时对象管理，避免把公共对话历史直接暴露给所有 Agent。通信通道覆盖点对点、主题、群组和运行时控制等协作关系；Agent 发送消息时提交结构化元数据、短摘要和资料引用。运行时根据接收者、主题、角色、权限、任务阶段和过滤条件选择性投递，避免无关消息进入不相关 Agent 的上下文。

### 3.2 验收指标

#### 3.2.1 软件验收条件

**一套面向大规模多智能体的运行时**

- 交付件形态：项目原型及源码
- 运行环境：基于业界主流服务器和 Linux 操作系统
- 功能描述：支持拆解用户任务并组织成多个 Agent 的协作流程由 Agent 运行时执行，Agent 运行时负责调度各 Agent 的调度顺序、协调它们的共享资源并在它们之间传递数据与结果。
- 验收标准：
  - 阶段 1.1：Agent 运行时可启动，能把多个 Agent 作为统一管理的对象来驱动运行
  - 阶段 1.2：支持多个 Agent 按任务依赖关系协作，上游完成后结果传给下游并触发调度运行
  - 阶段 2.1：支持 Agent 间通信以及上下文隔离，避免无关信息互相干扰。
  - 阶段 2.2：支持多个任务流程同时运行，并由 Agent 运行时在多个并行 Agent 之间协调调度、通信。相比于 naive 系统，本工作任务吞吐指标提升 10%，无关上下文注入率降低 10%
